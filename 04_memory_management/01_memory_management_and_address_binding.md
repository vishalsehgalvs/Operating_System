# Memory Management and Address Binding

> Memory management is how the OS allocates, tracks, and protects RAM for every running process; address binding is the specific mechanism that maps the logical addresses a program uses to the actual physical addresses in RAM — and this mapping can happen at compile time, load time, or during execution.

## 1. What Is Memory Management?

**Memory management** is the OS function that controls which process gets RAM, how much, and when it gets released.

**Parking lot analogy:**

```
  RAM = parking lot with limited spaces
  OS  = parking attendant

  Responsibilities:
  ┌─────────────────────────────────────────────────────────┐
  │  Allocate    → assign a free spot when a car arrives    │
  │  Track       → know which spots are occupied and by who │
  │  Deallocate  → free the spot when the car leaves        │
  │  Protect     → stop car A from parking in car B's spot  │
  │  Share       → allow carpooling in designated spots      │
  └─────────────────────────────────────────────────────────┘
```

**Main goals:**

1. Allocate memory to processes when needed
2. Track which memory is in use and by whom
3. Deallocate memory when processes finish
4. Protect each process's memory from other processes
5. Allow controlled sharing when necessary

## 2. Types of Addresses

Three kinds of addresses appear in memory management:

### Logical Address (Virtual Address)

Generated by the **CPU** when a program runs. Does not correspond to any real RAM location — it's the address the program believes it is using.

> Like a letter address: "123 Main St" is meaningful to people but must be converted to GPS coordinates before a mail carrier can physically find it.

### Physical Address

The **actual location** in RAM. This is what the memory hardware uses to read/write data.

> The GPS coordinates — the real, hardware-level location.

### Relative Address (Offset)

An address expressed as **distance from a base point**, not an absolute location.

> "Third shelf from the bottom" — relative to the floor (base), not an exact measurement.

### Summary Table

| Address Type     | Generated By          | Scope            | Example                  |
| ---------------- | --------------------- | ---------------- | ------------------------ |
| Logical Address  | CPU / Program         | Per-process view | Variable X at offset 100 |
| Physical Address | Memory hardware / MMU | Actual RAM       | RAM cell 14346           |
| Relative Address | Compiler / Assembler  | Offset from base | 50 bytes from start      |

## 3. What Is Address Binding?

**Address binding** is the process of **mapping logical addresses to physical addresses**.

**Hotel room analogy:**

```
  Online booking  = logical address  ("I booked a double room")
  Check-in        = address binding  (receptionist assigns room 412)
  Room 412        = physical address (actual physical room)

  Binding can happen:
    - At booking time   → compile-time binding (room locked in advance)
    - At check-in time  → load-time binding    (assigned on arrival)
    - Any time you move → execution-time binding (room can change during stay)
```

---

## 4. Types of Address Binding

### Compile-Time Binding

Physical addresses are embedded in the program **at compilation**. The program must always load at the same fixed location.

```
  Compiler decides: "Variable x lives at RAM address 2000"

  INT x;  // Always at physical address 2000
  INT y;  // Always at physical address 2004

  → Cannot move the program without recompiling
```

| Characteristic | Detail                                       |
| -------------- | -------------------------------------------- |
| When           | At compile time                              |
| Flexibility    | None — fixed address required                |
| Speed          | Fastest (no runtime translation)             |
| Used in        | Simple embedded systems, bare-metal programs |

---

### Load-Time Binding

The compiler generates **relocatable code** (using offsets from 0). The **loader** translates these to physical addresses when the program is loaded into RAM.

```
  Compiler output (relocatable):
    Variable x  → offset 0   (relative to wherever we load)
    Variable y  → offset 4

  At load time (program placed at address 3000):
    x lives at: 3000 + 0 = 3000
    y lives at: 3000 + 4 = 3004

  Next run (placed at address 5000):
    x lives at: 5000 + 0 = 5000
    y lives at: 5000 + 4 = 5004
```

| Characteristic | Detail                                         |
| -------------- | ---------------------------------------------- |
| When           | When program is loaded into RAM                |
| Flexibility    | Different location each run; fixed once loaded |
| Speed          | Slight overhead at load time only              |
| Limitation     | Cannot move the process after it starts        |

---

### Execution-Time (Runtime) Binding

Logical-to-physical translation happens **on every memory access**, during execution. Requires hardware support (MMU). Process can be **moved in RAM while running**.

```
  Program always uses logical addresses:
    Logical address of x = 100
    Logical address of y = 104

  MMU translates using base register at runtime:
    Base register = 4000
    Physical x = 4000 + 100 = 4100
    Physical y = 4000 + 104 = 4104

  Process moves to a different location:
    Base register = 7000
    Physical x = 7000 + 100 = 7100
    Physical y = 7000 + 104 = 7104

  Program code never changes — only the base register changes!
```

| Characteristic  | Detail                                     |
| --------------- | ------------------------------------------ |
| When            | On every memory access during execution    |
| Flexibility     | Maximum — process can be relocated anytime |
| Hardware needed | MMU (Memory Management Unit)               |
| Used in         | All modern general-purpose OSes            |

### Comparison

| Binding Type   | When It Happens        | Can Process Move?  | Modern Usage             |
| -------------- | ---------------------- | ------------------ | ------------------------ |
| Compile-Time   | During compilation     | No                 | Rare (embedded systems)  |
| Load-Time      | When program loads     | No (after loading) | Occasional               |
| Execution-Time | On every memory access | Yes (anytime)      | Common — all modern OSes |

## 5. Memory Management Unit (MMU)

The **MMU** is a hardware component (usually inside the CPU) that **automatically translates every logical address to a physical address** before it reaches RAM.

```
  CPU generates                  MMU translates               Physical RAM
  logical address                  (hardware)                  is accessed

  CPU ──[logical: 346]──► MMU ──[physical: 14346]──► RAM[14346]
                           │
                     Uses Base and
                     Limit registers
```

**How MMU works with base + limit registers:**

```c
// Hardware logic inside MMU

Base  Register = 14000   // Where this process starts in physical RAM
Limit Register = 3000    // Size of this process's memory

// CPU sends logical address 346:
Step 1: Is 346 < 3000 (limit)?  → YES, valid
Step 2: Physical = Base + Logical = 14000 + 346 = 14346
Step 3: Send 14346 to RAM → data returned

// CPU sends logical address 3500:
Step 1: Is 3500 < 3000 (limit)?  → NO, out of bounds!
Step 2: Raise hardware trap → OS handles error (segmentation fault)
```

**MMU as GPS:**

```
  Program says "go to address 346"  (logical — relative to process)
  MMU maps it to "RAM cell 14346"   (physical — absolute location)

  Like GPS: type "Coffee Shop" → GPS converts to real-world coordinates
```

## 6. Logical vs Physical Address Space

### Logical Address Space

The **complete set of logical addresses** a process can generate. Each process has its own logical address space, typically starting from 0.

> Like each person numbering their own books: "book 1, book 2, book 3." Two people can both have a "book 1" — no conflict because they're different people's lists.

### Physical Address Space

The **actual RAM** — shared by all processes. Limited by how much RAM is installed.

> The actual physical library shelves — shared, finite, everyone's books must fit without overlap.

```
  TWO PROCESSES, SAME LOGICAL ADDRESSES, DIFFERENT PHYSICAL LOCATIONS:

  Process A (base = 10000):       Process B (base = 20000):
  Logical 0 → Physical 10000     Logical 0 → Physical 20000
  Logical 4 → Physical 10004     Logical 4 → Physical 20004
  Logical 8 → Physical 10008     Logical 8 → Physical 20008

  Both use logical address 0, but they map to different physical locations.
  No conflict. Each process has the ILLUSION of owning memory from 0.
```

## 7. Swapping

**Swapping** moves a process temporarily from RAM to disk (swap space), freeing physical memory for other processes. When needed again, it is brought back into RAM.

**Restaurant analogy:**

```
  Tables = RAM slots
  Lounge = disk (swap space)

  Peak hours: table full → waiting group moved to lounge
  Their food arrives → brought back to a table (possibly different one!)
```

```
  Timeline:
  t=0:  Process A loaded at physical 5000
  t=1:  Process A waits for I/O → swapped OUT to disk, RAM at 5000 freed
  t=2:  Process C needs to run → loaded at 5000
  t=3:  Process A's I/O done → swapped IN at 12000 (different location!)

  Requires execution-time binding — process runs from new location without code changes!
```

**Why swapping requires execution-time binding:** The process may come back at a different address. If addresses were compiled in (compile-time) or fixed at load time, the process would fail after swapping to a new location.

## 8. Contiguous Memory Allocation

Each process occupies a **single continuous block** of RAM. The entire process must fit in one uninterrupted chunk.

```
  Physical RAM:
  ┌──────────────────────────────────────────────────────┐
  │  OS (reserved)  │  Process A  │  Free  │  Process B  │
  │  0 – 4KB        │  4KB – 10KB │ 10–12KB│  12KB – 16KB│
  └──────────────────────────────────────────────────────┘

  Process A: one contiguous block from 4KB to 10KB
  Process B: one contiguous block from 12KB to 16KB
  Free hole:  one contiguous block from 10KB to 12KB
```

**Limitation:** Over time, free memory becomes split into small pieces that are too small for any new process — this is called **fragmentation** (covered in the next topic).

## 9. Memory Protection

The OS ensures no process can read or write another process's memory — using the MMU's base + limit registers.

```c
// Process P1 allocation:
Base  = 10000
Limit = 2000    // P1 can only access 10000 to 11999

// Process P2 allocation:
Base  = 15000
Limit = 3000    // P2 can only access 15000 to 17999

// P1 accesses logical address 500:
Physical = 10000 + 500 = 10500
10500 < 10000 + 2000 (= 12000)? YES → allowed ✓

// P1 tries to access logical address 3000 (outside its range):
Physical = 10000 + 3000 = 13000
13000 < 12000? NO → hardware trap → OS terminates P1 with SIGSEGV
```

Memory protection prevents:

- Buggy programs from corrupting other processes' data
- Malicious programs from reading sensitive data from other processes
- Any process from modifying OS kernel memory

## 10. Allocation Tracking and Deallocation

The OS tracks free and used memory using one of two data structures:

### Bitmap

Each bit represents one fixed-size memory unit (e.g., 1 KB). `0` = free, `1` = in use.

```
  Memory units:  [P1][P1][-- ][-- ][P2][P2][P2][-- ]
  Bitmap:          1   1   0   0   1   1   1   0

  Easy to scan for free space, but slow for large memories (many bits to scan).
```

### Linked List

Each node describes one segment: `[start address, size, status (free/process)]`

```
  Head → [0, 2KB, P1] → [2KB, 2KB, Free] → [4KB, 3KB, P2] → [7KB, 1KB, Free] → NULL
```

### Coalescing (Merging Free Blocks)

When a process frees memory, adjacent free blocks are merged into one larger block:

```
  Before:     [P1 2KB] → [Free 1KB] → [Free 1KB] → [P2 2KB]
  Coalesce:   [P1 2KB] → [Free 2KB]               → [P2 2KB]

  After P1 terminates:
  Before:     [Free 2KB] → [Free 2KB] → [P2 2KB]
  Coalesce:   [Free 4KB]               → [P2 2KB]

  One large 4KB block is much more useful than two separate 2KB blocks!
```

## 11. Real-World Walkthrough

**Setup:** 16 KB of RAM, three programs launched sequentially.

```c
// Step 1: Browser starts (needs 4 KB)
Browser: logical 0–4095 → physical 0–4095
  Base = 0, Limit = 4096
RAM: [0–4KB: Browser]  [4KB–16KB: Free]

// Step 2: Text editor starts (needs 2 KB)
Editor: logical 0–2047 → physical 4096–6143
  Base = 4096, Limit = 2048
RAM: [0–4KB: Browser]  [4KB–6KB: Editor]  [6KB–16KB: Free]

// Step 3: Music player starts (needs 3 KB)
Player: logical 0–3071 → physical 6144–9215
  Base = 6144, Limit = 3072
RAM: [0–4KB: Browser]  [4KB–6KB: Editor]  [6KB–9KB: Player]  [9KB–16KB: Free]

// Editor accesses variable at logical address 100:
  MMU: Physical = 4096 + 100 = 4196
  Check: 4196 < 4096 + 2048 (= 6144)? YES → allowed ✓

// Editor accesses logical address 3000 (outside its 2KB limit):
  MMU: Physical = 4096 + 3000 = 7096
  Check: 7096 < 6144? NO → hardware trap → segmentation fault!

// Step 4: Editor closes
RAM: [0–4KB: Browser]  [4KB–6KB: Free]  [6KB–9KB: Player]  [9KB–16KB: Free]
OS can now allocate the 2KB hole to a new process.
```

## 12. Key Takeaways

- **Memory management** allocates, tracks, protects, and deallocates RAM for each running process
- Three address types: **Logical** (program's view) → **Physical** (actual RAM) → **Relative** (offset from base)
- **Address binding** = mapping logical addresses to physical addresses; can happen at:
  1. **Compile time** — fixed location, fastest, inflexible (embedded systems)
  2. **Load time** — assigned at load, fixed after that (occasional use)
  3. **Execution time** — translated on every access, maximum flexibility (all modern OSes)
- The **MMU** (inside CPU) performs translation using `Base` and `Limit` registers — every memory access goes through it
- Each process gets its own **logical address space** starting at 0; the MMU maps it to its actual physical location
- **Swapping** moves processes to/from disk — requires execution-time binding so the process can run from a new location
- **Memory protection** via limit register: any logical address ≥ limit → hardware trap → OS handles error
- Free memory is tracked via **bitmaps** (per-unit bits) or **linked lists** (segment records)
- **Coalescing** merges adjacent free blocks to prevent memory from being fragmented into many tiny unusable pieces
